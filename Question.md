## Library management system
1. Define the fundamental Book entity containing attributes like ISBN, title, and author. Require all instances to be explicitly initialized with these attributes upon creation to prevent empty or invalid records.
2. Protect a Member entity's sensitive data, such as borrowedBooksCount and fineBalance. Ensure these fields cannot be modified directly from outside the class, requiring state changes to occur only through validated transaction methods like processBorrow() or payFine().
3. Create an abstract system blueprint called LibraryAsset that forces any concrete item in the system to implement standard operational methods like checkout() and returnItem(), completely hiding the internal database transaction logic from the user.
4. Establish a DigitalAsset class that inherits standard identification attributes (ID, title) from a base LibraryAsset class, but introduces digital-specific attributes like fileSizeMB and downloadUrl.
5. Extend a base User class into distinct StudentMember and FacultyMember classes to establish different default privileges, such as varying maximum checkout limits and borrowing duration windows.
6. Design a calculateLateFee(daysLate) method in the base class, and override it in derived classes to compute fees dynamically based on the item type (e.g., standard books charge 1 per day, while high-demand reserve materials charge 5 per day).
7. Implement multiple search() functionalities within the Catalog class, allowing the system to query the database using different parameter signatures: a string for the title, a string for the author, or an integer for the exact ISBN.
8. Define an IReservable interface containing a placeHold() method signature. Apply this interface strictly to physical assets like HardcoverBook or LibrarianTablet, ensuring digital items that have infinite copies cannot logically be placed on physical hold.
9. Implement a global tracking mechanism using a totalActiveLoans variable. This integer must increment globally across the entire system whenever any item is checked out, independent of any specific book or user object instance.
10. Model the strict structural relationship between a LibraryFacility and its Sections (e.g., Reference Room, Children's Area). Ensure that if the central LibraryFacility instance is closed or destroyed in the system, its dependent Section objects are strictly destroyed as well.
